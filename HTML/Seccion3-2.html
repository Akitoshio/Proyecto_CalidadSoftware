    Modelos Tradicionales

    Los modelos tradicionales de SDLC ofrecen diferentes enfoques para estructurar proyectos como el Modelo en Cascada, Modelo Espiral, Modelo Incremental, V-Model, Modelo Prototipado y Modelo Iterativo.Estos modelos siguen siendo relevantes en entornos donde la predictibilidad es prioritaria, como proyectos gubernamentales o normativos.

    Modelo Cascada

    Es uno de los modelos tradicionales de desarrollo de software, que se caracteriza por su enfoque secuencial y lineal. Por ende en este modelo cada fase debe completarse mas antes de pasar a la siguiente fase, por lo cual normalmente no se regresa a fases anteriores (aunque en la práctica a veces se hacen ajustes).

    Etapas:

    Definicion de Requisitos:Se reúnen y documentan todas las necesidades del cliente.

    Analisis del sistema:Se estudian los requisitos para determinar la viabilidad técnica y se modelan procesos y datos.

    Diseño:Se crea la arquitectura del sistema: bases de datos, interfaces, diagramas, flujos de datos.

    Codificacion:Los programadores escriben el código de acuerdo con el diseño aprobado.

    Prueba:Se verifica y valida que el software funcione correctamente y cumpla los requisitos.

    Despliegue:Se instala el sistema en el entorno real y se capacita a los usuarios.

    Mantenimiento:Se corrigen errores, se realizan mejoras o adaptaciones a nuevos requisitos.

    Ventajas:

    Descripción: claridad de fases, documentación sólida, cumplimiento normativo.

    Desventajas:

    Descripción:inflexible ante cambios, riesgos detectados tarde.

    Casos de uso:

    Descripción: proyectos con requisitos estables, altamente regulados.

    Modelo Espiral

    Es un modelo de desarrollo de software iterativo y evolutivo, que combina elementos del modelo en cascada con un enfoque principal que es identificar y minimizar riesgos durante todo el desarrollo del proyecto.

    Etapas:

    Determinacion de Objetivos y Planificación:Se definen los objetivos del ciclo y se identifican los requisitos iniciales.

    Análisis de riesgos:Se identifican posibles riesgos técnicos, de costos o de tiempo y se planifican estrategias para mitigarlos.

    Desarrollo o implementación:Se construye un prototipo o versión del software que puede ser probado y evaluado.

    Evaluación del cliente:El usuario evalúa el prototipo o versión desarrollada, proporcionando retroalimentación que guiará la siguiente iteración.

    Ventajas:

    Descripción:evalúa riesgos explícitamente, permite prototipos, flexible.

    Desventajas:

    Descripción:complejo de gestionar, costoso.

    Casos de uso:

    Descripción:proyectos innovadores, críticos y de gran escala.

    Modelo Incremental

    Es un enfoque de desarrollo de software en el que el sistema se construye y entrega en partes o incrementos, por lo cual cada incremento agrega nuevas funcionalidades al software existente hasta completar el producto final. Este modelo combina desarrollo por fases con la entrega temprana de partes funcionales, lo que permite a los usuarios probar y usar parcialmente el sistema mientras se sigue desarrollando.

    Etapas:

    Planificación de requisitos:Se identifican los requisitos generales del sistema y se definen prioridades para los incrementos.

    Análisis y diseño de cada incremento:Se analiza y diseña la funcionalidad que se incluirá en el incremento actual.

    Implementación o codificación del incremento:Se desarrolla la parte funcional del software correspondiente a ese incremento.

    Pruebas del incremento:Se realizan pruebas unitarias e integradas para asegurar que el incremento funciona correctamente.

    Entrega del incremento:El incremento se entrega al usuario para su evaluación y retroalimentación.

    Iteración de nuevos incrementos:Se repiten las etapas anteriores para cada nuevo incremento hasta completar el sistema final.

    Ventajas:

    Descripción:feedback temprano, entrega progresiva, menos riesgo de fallo tota

    Desventajas:

    Descripción:requiere arquitectura modular, riesgo de deuda técnica.

    Casos de uso:

    Descripción:sistemas grandes que se construyen por etapas.

    V-Model

    Es un modelo de desarrollo de software basado en la verificación y validación. Por lo cual se divide en dos parte, la parte izquierda representa la definición y diseño, mientras que la parte derecha representa la validación y pruebas.

    Etapas:

    Requisitos del Sistema: Se identifican y documentan los requisitos del usuario (aqui tambien se asocia la prueba de aceptacion de usuario).

    Requisitos Funcionales y de Software: Se especifica qué debe hacer el sistema y cómo debe comportarse (aqui se asocia la prueba de sistema).

    Diseño arquitectonico: Se define la arquitectura general del software y cómo interactuarán los módulos (aqui se asocia la prueba de integracion).

    Diseño detallado: Se diseñan los módulos individuales, sus interfaces y algoritmos (aqui tambien se asocia la pruena unitaria).

    Validacion y Pruebas: Aqui se realizan las pruebas ya mensionadas correspondiente en cada etapa que son: Prueba Unitaria (verifica cada modulo funcione correctamente), Prueba de integracion(verifica que cad modulo funcion bien juntos), Prueba del Sistema (verifica que el sistema cumpla con los requisitos funcionales) y Prueba de aceptacion (verifica que el software cumpla con las necesidades del usuario final)

    Ventajas:

    Descripción:trazabilidad clara entre requisitos y pruebas

    Desventajas:

    Descripción:: rigidez, requiere pruebas definidas tempranamente.

    Casos de uso:

    Descripción:sectores donde V&V; son críticos (farmacéutica, aeroespacial).

    Modelo Prototipado

    Es un enfoque del desarrollo de software en el que se construye una versión preliminar o “prototipo” del sistema antes de desarrollar el producto final. Este prototipo sirve para que los usuarios y desarrolladores puedan visualizar, probar y refinar las funcionalidades del software antes de su implementación definitiva, por lo cual su objetivo principal es reducir riesgos y errores, asegurando que el sistema cumpla con las expectativas y necesidades del usuario.

    Etapas:

    Recolección de requisitos:Se identifican las necesidades del usuario y qué funcionalidades debe tener el sistema.

    Desarrollo del prototipo inicial:Se construye una primera versión simplificada del software que incluya las funciones principales.

    Evaluación por parte del usuario:El usuario prueba el prototipo y da retroalimentación sobre qué mejorar o cambiar.

    Refinamiento del prototipo:Se realizan ajustes y se crean nuevas versiones basadas en los comentarios del usuario.

    Desarrollo del producto final:Una vez aprobado el prototipo, se construye el software completo y funcional siguiendo buenas prácticas de codificación.

    MantenimientoDespués de la implementación final, se realizan ajustes o mejoras según sea necesario.

    Ventajas:

    Descripción:aclara expectativas, mejora experiencia de usuario.

    Desventajas:

    Descripción: puede generar falsas expectativas si prototipo se confunde con producto final.

    Casos de uso:

    Descripción: proyectos con requisitos poco claros o alto foco en UX

    Modelo Iterativo

    Este modelo es un enfoque de desarrollo de software en el que el sistema se desarrolla en ciclos o iteraciones, por lo cual cada iteración produce una versión parcial del software que se mejora progresivamente hasta alcanzar el producto final. Este modelo a diferencia del modelo en cascada, nos permite revisar y modificar los requisitos durante el desarrollo, lo que hace que sea más flexible y adaptable a cambios.

    Etapas:

    Planificación:Se determinan los objetivos de la iteración y qué funcionalidades se implementarán.

    Analisis y Diseño:Se analiza y diseña la funcionalidad que se desarrollará en la iteración actual.

    Implementación/codificacion:Se desarrolla el código correspondiente a la iteración.

    Pruebas:Se realizan pruebas unitarias, de integración y del sistema para asegurar que la funcionalidad implementada funciona correctamente.

    Evaluación/Retroalimentación del usuario:El usuario prueba la versión parcial del software y da comentarios que guiarán la siguiente iteración

    Repeticion del ciclo:Se inicia una nueva iteración incorporando mejoras, nuevas funcionalidades o cambios según la retroalimentación.

    Ventajas:

    Descripción:adaptación y mejora continua

    Desventajas:

    Descripción: riesgo de iteraciones interminables si no se controla.

    Casos de uso:

    Descripción:proyectos que requieren refinamiento progresivo
