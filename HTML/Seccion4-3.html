    Extreme Programming (XP)

    es una metodología ágil enfocada en mejorar la calidad del software y la capacidad de respuesta ante cambios. Se centra en programación colaborativa, retroalimentación rápida y prácticas de ingeniería de alta calidad, ideal para entornos de requisitos cambiantes.

    Contenidos

    Principios de XP

    Comunicación: Fomentar la colaboración constante entre desarrolladores, cliente y todo el equipo.

    Simplicidad: Construir solo lo necesario hoy, evitando complejidad futura innecesaria.

    Retroalimentación temprana: Detectar y corregir errores lo antes posible mediante pruebas continuas y entregas frecuentes.

    Coraje (Valentía): Estar dispuesto a refactorizar, eliminar código o cambiar el diseño cuando sea necesario.

    Respeto: Todos en el equipo valoran el trabajo de los demás y comparten la responsabilidad del proyecto.

    Roles

    Cliente o Cliente On-Site: Representa al usuario final, prioriza historias de usuario y responde preguntas del equipo.

    Desarrolladores (Programmers): Construyen el software, escriben pruebas y refactorizan el código.

    Coach: Ayuda al equipo a seguir los valores y prácticas de XP.

    Tracker: Mide el progreso y ayuda a ajustar las estimaciones.

    Tester: Apoya en la definición de pruebas de aceptación junto al cliente.

    Gestor (Manager): Elimina impedimentos y facilita recursos.

    Ceremonias

    Planning Game (Planificación): El cliente y el equipo priorizan historias de usuario y planifican la siguiente iteración.

    Iteraciones cortas: Ciclos de 1–2 semanas con entregas funcionales.

    Stand-up Meetings: Reuniones diarias breves para sincronizar el trabajo.

    Pair Programming: Dos desarrolladores programan juntos en un solo equipo/estación.

    Continuous Integration: Integrar y probar el código varias veces al día.

    On-site Customer Collaboration: Contacto continuo con el cliente para dudas y validación.

    Retrospectivas: Revisión de la iteración para mejorar procesos.

    Artefacto

    Historias de Usuario: Descripciones simples de funcionalidades desde la perspectiva del cliente.

    Tarjetas de Historia (Story Cards): Fichas físicas o digitales para planificar y estimar.

    Pruebas Unitarias y de Aceptación: Base del desarrollo guiado por pruebas (TDD)

    Código Fuente Refactorizado: El código limpio y mantenible es un producto clave

    Gráficos de progreso/velocidad: Para seguimiento de iteraciones.
